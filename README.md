## 先搞懂：端口被封 ≠ IP 被墙

这是最容易搞混的地方。

**IP 被墙**：国内 ICMP（ping）不通，TCP 所有端口都没响应，海外能 ping 通但国内完全连不上。这种才是"墙"。

**端口被封**：IP 可以 ping 通，甚至 SSH 22 端口（或其他端口）还好的，只是某个特定端口在国内不可达。典型场景：科学上网工具用的那个监听端口被针对了。

判断方法很简单——用在线端口检测工具，分别从国内和国外测试你的 IP 和对应端口：

- 国内不通、国外通 → 端口被封（或 IP 被墙）
- 只有某几个端口国内不通，其他端口正常 → **只是端口问题**
- 所有端口国内都不通，ICMP 也不通 → IP 被完全封锁

这个判断关系到后面怎么处理，不能跳过。

## DMIT 端口被封：三种应对思路

### 思路一：换一个端口

最快的方法。端口只是你服务监听的入口，改一个新端口让服务重新跑起来，代价几乎为零。

操作上：修改你服务端配置里的监听端口，然后客户端同步改掉，重启服务就行。注意改完之后用同样的检测工具确认新端口在国内是通的再投入使用。

有一点要注意：短时间内连续换多个端口被封，说明流量特征本身可能被识别了，这时候不建议继续无脑换端口，应该考虑换协议或加混淆。

### 思路二：端口跳转（iptables 转发）

还是用原来的端口监听服务，但通过 iptables 把另一个端口的流量转发进来。客户端连新端口，服务端实际上走的是原来那个端口。

命令参考（把 40000-50000 的流量转发到 443）：

bash
iptables -t nat -A PREROUTING -p tcp --dport 40000:50000 -j REDIRECT --to-ports 443


这样客户端可以用 40000 到 50000 之间的任意端口连接，本机实际上还是走 443，netstat 也看不到那些端口在监听。端口被封了就随时换一个新的范围，比每次都改配置文件省事。

### 思路三：协议加伪装

说实话，如果端口三番五次被封，换端口只是在和检测系统打游击。更稳的做法是把流量包装成正常 HTTPS 请求，让墙从特征上就识别不出来。这方面 TLS 伪装相关方案比裸跑协议抗封性强得多，有需要的话可以单独研究一下。
